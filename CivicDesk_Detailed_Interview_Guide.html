Java Core & OOP

Plain Java building blocks visible across entity, service, dto and enums packages.

#ConceptTheory + Why we use itExact file & how it is used
1Class / Object / Constructor A class is a blueprint; an object is an instance; constructors initialise state. Why: constructors give us a single, valid way to build objects (used for DI too). module/iam/entity/User.java — new User() in service; UserController.java uses constructor injection: UserController(UserService, AuditService). What is the difference between a class and an object?
2Encapsulation Private fields exposed via getters/setters. Why: protects invariants and hides the password hash from callers. All fields private in entity/User.java; dto/response/UserResponse.java deliberately has no passwordHash field — “never exposes the password hash”.
3Inheritance Child reuses a parent (IS-A). Why: reuse framework behaviour and standard exception semantics. common/exception/ResourceNotFoundException.java extends RuntimeException; security/JwtAuthFilter.java extends OncePerRequestFilter.
4Polymorphism One reference type, many implementations (runtime overriding). Why: code against interfaces, swap implementations freely. service/UserService.java (interface) reference points to UserServiceImpl.java; @Override on every impl method.
5Abstraction Expose what, hide how, via interfaces/abstract types. Why: controller need not know the DB or business details. service/UserService.java, service/AuthService.java interfaces; JpaRepository abstracts all SQL.
6Interface vs Abstract class Interface = pure contract (multiple); abstract class = partial base. Why: Spring beans are wired by interface for testability. repository/UserRepository.java is an interface extending JpaRepository + JpaSpecificationExecutor — Spring generates the impl.
7Enums Fixed set of named constants, optionally with behaviour. Why: roles/statuses are a closed set — type-safe, no magic strings. module/iam/enums/Role.java (CIT, FO, DS, ENG, CO, ADM); enums/UserStatus.java with .getLabel() / normalize().
8Collections (List/Map/Set) Reusable containers for groups of objects. Why: return many users; declare immutable allow-lists. UserServiceImpl.java: List.of(Role.FO, Role.ENG, Role.CO) (immutable); JwtAuthFilter.java: List.of(new SimpleGrantedAuthority(...)).
9Exception handling try/catch + custom exceptions model failures. Why: turn domain errors into correct HTTP codes; never crash the caller. auditlog/client/AuditClient.java catches RuntimeException and logs (audit is best-effort). 11 custom exceptions in common/exception/.
10Generics Parameterised types checked at compile time. Why: type-safe repositories and responses, no casting. JpaRepository<User, String>, ResponseEntity<ApiResponse>, PageResponse<UserResponse>, Optional<User>.
11Annotations Metadata that frameworks read to add behaviour. Why: declarative wiring/validation/mapping instead of boilerplate code. Everywhere: @Entity, @Service, @RestController, @Valid, @Transactional, @Test.
12Java 8+ (Stream / Lambda / Optional) Declarative data processing; Optional = null-safe box. Why: readable transforms and explicit “might be absent”. config/DataSeeder.java findAll().stream().map(...).filter(...).mapToInt(...).max(); UserServiceImpl.java findById(id).orElseThrow(...). Why prefer Optional over returning null?

Spring Core & Dependency Injection

The container creates and wires every bean; this project uses constructor injection exclusively.

#ConceptTheory + Why we use itExact file & how it is used
1IoC Container Spring (not you) instantiates, wires and manages objects. Why: loose coupling, central lifecycle, easy testing. Bootstrapped by CivicDeskApplication.java → SpringApplication.run(...).
2Constructor Injection Dependencies passed via constructor into final fields. Why: immutable, non-null deps, trivially mockable in tests. UserServiceImpl.java: UserServiceImpl(UserRepository, DepartmentRepository); same pattern in every *Impl / controller. No field @Autowired used.
3Stereotypes @Service / @Repository / @RestController / @Component / @Configuration mark beans by role. Why: component-scan picks them up; intent is explicit. @Service on UserServiceImpl.java; @Repository on UserRepository.java; @Component on util/JwtUtil.java, DataSeeder.java.
4@Configuration & @Bean Java config class; @Bean registers a manually built object. Why: wire framework objects we don’t own (encoder, filter chain). config/SecurityConfig.java: @Bean SecurityFilterChain, @Bean PasswordEncoder → BCryptPasswordEncoder(10); CorsConfig.java, OpenApiConfig.java.
5Bean scope / Singleton Beans are singletons by default — one shared instance. Why: stateless services are cheap and thread-safe to share. All services/repos/utils are singletons managed by the container.
6CommandLineRunner Runs code once after the context is ready. Why: seed the first ADM + departments before any request arrives. config/DataSeeder.java implements CommandLineRunner → run() seeds data.

Maven (Build & Dependencies)

Single source of truth: pom.xml at the project root.

#ConceptTheory + Why we use itExact file & how it is used
1Parent POM Inherit managed versions from spring-boot-starter-parent. Why: one Boot version (3.4.1) aligns all dependency versions. pom.xml → <parent>spring-boot-starter-parent 3.4.1</parent>, <java.version>21</java.version>.
2Starters Curated dependency bundles. Why: add a capability without hunting individual jars/versions. starter-web, starter-security, starter-data-jpa, starter-validation, starter-test.

4Build lifecycle clean → compile → test → package → install. Why: repeatable builds; tests gate packaging. mvn clean package builds the JAR; mvn test runs all JUnit tests; wrapper in .mvn/wrapper/ (./mvnw).
5Plugin Extends the build. Why: produce a runnable “fat” JAR with embedded Tomcat. pom.xml → spring-boot-maven-plugin in <build><plugins>.

Spring Boot 3

Auto-configuration + externalised config. Server runs on :8081 with context path /civicDesk.

#ConceptTheory + Why we use itExact file & how it is used
1@SpringBootApplication = @Configuration + @EnableAutoConfiguration + @ComponentScan. Why: one annotation boots the whole app. CivicDeskApplication.java — main() calls SpringApplication.run(CivicDeskApplication.class, args).
2Auto-configuration Beans configured from classpath + properties. Why: DataSource, JPA, Tomcat, Security wired with near-zero code. Adding starter-data-jpa + MySQL props auto-creates the DataSource & EntityManager. How does Boot decide which beans to auto-configure?
3Embedded Tomcat Servlet container inside the JAR. Why: java -jar runs the API — no external server. Provided by starter-web; port/context from properties.
4application.properties Externalised configuration. Why: change DB/port/secret without recompiling. resources/application.properties — datasource URL, server.port=8081, server.servlet.context-path=/civicDesk, JWT, CORS.
5Profiles Environment-specific config. Why: separate dev/prod/test settings. application-dev.properties, application-prod.properties, test/resources/application-test.properties (H2).
6@Value Inject a single property. Why: pull secrets/admin seed values from config. DataSeeder.java @Value("${app.admin.email}"); AuthServiceImpl.java @Value("${app.jwt.expiry}").
7@ConfigurationProperties Bind a group of properties to a typed bean. Why: group related JWT settings cleanly. config/JwtConfig.java binds app.jwt.* (secret, expiry) used by JwtUtil.java.
8Env-var defaults ${VAR:default} placeholder syntax. Why: 12-factor config — override by env in real deployments. application.properties → app.admin.email=${ADMIN_EMAIL:admin@civicdesk.gov}.

Spring REST API

Controllers live in module/iam/controller; all responses wrapped in the ApiResponse envelope.

#ConceptTheory + Why we use itExact file & how it is used
1REST principles Stateless, resource URIs, correct HTTP verbs/status. Why: predictable, cacheable, scalable API; JWT keeps it stateless. SessionCreationPolicy.STATELESS in SecurityConfig.java; resources like /iam/users/{id}/status. Name the key REST constraints.
2@RestController @Controller + @ResponseBody — returns JSON. Why: it’s a JSON API, not server-rendered HTML. controller/UserController.java, AuthController.java, DepartmentController.java.
3@RequestMapping + verb mappings Base path + @GetMapping/@PostMapping/@PutMapping. Why: map each verb to the right action. UserController.java @RequestMapping("/iam/users"); @GetMapping("/me"), @PostMapping, @PutMapping("/{id}/status").
4@PathVariable / @RequestParam / @RequestBody Bind URL segment / query / JSON body. Why: cleanly map HTTP input to Java params. UserController.java: @PathVariable String id; @RequestParam(defaultValue="0") int page; @Valid @RequestBody CreateUserRequest.
5ResponseEntity + status codes Full control of body + HTTP status. Why: return 201 on create, 200 on read, etc. UserController.java: ResponseEntity.status(201).body(ApiResponse.of("User created...", null)), ResponseEntity.ok(ApiResponse.data(user)).
6DTO pattern Separate request/response objects from entities. Why: never leak entity internals (e.g. password hash); shape the contract. Requests in dto/request/ (CreateUserRequest, StaffLoginRequest…); responses in dto/response/ (UserResponse via UserResponse.from(user)).
7Response envelope Consistent {message?, data?} shape. Why: frontend parses every response uniformly. common/response/ApiResponse.java — data() / of() / error() factory methods; @JsonInclude(NON_NULL).
8Global exception handling @RestControllerAdvice + @ExceptionHandler centralise errors. Why: one place maps each exception → HTTP status; controllers stay clean. common/exception/GlobalExceptionHandler.java — maps ResourceNotFound→404, DuplicateEmail→409, Forbidden→403, AccountSuspended→423, validation→400.
9Pagination response Page metadata + content. Why: never dump the whole table to the client. common/response/PageResponse.java — PageResponse.from(page, UserResponse::from) in UserServiceImpl.java.

Spring Data JPA & Hibernate

Entities in module/iam/entity; repositories are interfaces Spring implements at runtime.

Design note (important for viva): this project models relationships by ID reference (e.g. User.departmentId is a plain @Column, not a @ManyToOne to Department). Why: it keeps modules loosely coupled and split-ready (microservice-friendly), avoids lazy-loading/N+1 surprises, and keeps JSON flat. So @OneToMany/@ManyToOne/@JoinColumn are known concepts but intentionally not used here.
#ConceptTheory + Why we use itExact file & how it is used
1ORM Map Java objects ↔ DB rows; Hibernate generates SQL. Why: work with objects, not hand-written SQL. entity/User.java ↔ users table; Hibernate is the JPA provider (from starter-data-jpa).
2@Entity / @Table / @Index Mark a persistent class, its table and indexes. Why: indexes speed up the role/status filters we query on. entity/User.java: @Entity @Table(name="users", indexes={@Index(... role,status ...)}); entity/Department.java → departments.
3@Id / @GeneratedValue / custom generator Primary key + generation strategy. Why: human-friendly numeric string ids starting at 10000001. entity/User.java uses @GenericGenerator → common/id/NumericStringSequenceGenerator.java; Department.java uses assigned ids (DPT01…).
4@Column constraints Map fields with length/nullable/unique. Why: enforce DB-level integrity (unique email). entity/User.java: @Column(nullable=false, unique=true, length=150) email; updatable=false on id.
5@CreationTimestamp / @UpdateTimestamp Hibernate auto-fills audit timestamps. Why: track created/updated without manual code. entity/User.java: @CreationTimestamp createdAt, @UpdateTimestamp updatedAt.
6JpaRepository Ready CRUD: save/findById/findAll/deleteById. Why: no boilerplate DAO code. repository/UserRepository.java extends JpaRepository<User, String> — save(), findById() used in services.
7Derived query methods Spring builds SQL from method names. Why: express simple queries with zero SQL. UserRepository.java: findByEmail(), existsByEmail(), findByDepartmentIdAndStatus(...), findByRole(...).
8Specifications (dynamic queries) Composable Specification<T> predicates via Criteria API. Why: optional role+status+department filters without a method per combo. repository/spec/UserSpecifications.java (hasRole/hasStatus/inDepartment) combined with .and(...) in UserServiceImpl.getUsers(); repo extends JpaSpecificationExecutor<User>.
9@Transactional Method runs in one DB transaction (commit/rollback). Why: creating a user + updating the department must be atomic. UserServiceImpl.createUser() and AuthServiceImpl.register()/setPassword() are @Transactional. What triggers a rollback by default?
10Pagination & Sorting Pageable/Page fetch one page at a time. Why: performance — bounded result sets. UserServiceImpl.getUsers(): PageRequest.of(page,size) → userRepository.findAll(spec, pageable) → Page<User>.
11ddl-auto / dialect Hibernate schema generation + SQL dialect. Why: auto-evolve dev schema; emit MySQL-specific SQL. application.properties: spring.jpa.hibernate.ddl-auto=update, show-sql=true, dialect=MySQL8Dialect. Why is update unsafe in production?

MySQL & ANSI SQL

Hibernate generates the SQL; understanding what it emits is the viva focus. DB: civicdesk on localhost:3306.

#ConceptTheory + Why we use itExact file & how it is used
1DDL / DML / DQL Define schema / change data / read data. Why: Hibernate issues all three for us. DDL auto-generated from entity/*.java; DML/DQL from save()/findAll(). show-sql=true prints them.
2SELECT / INSERT / UPDATE Core data statements. Why: CRUD on users/departments. save(new User) → INSERT; setStatus()+save() → UPDATE (UserServiceImpl.updateStatus()); findByEmail → SELECT.
3WHERE / ORDER BY Filter and sort rows. Why: role/status/department filtering & paged ordering. Generated from UserSpecifications.java predicates + PageRequest sort.
4Primary Key Uniquely identifies a row. Why: stable identity for each user/department. @Id userId (User.java), @Id departmentId (Department.java).
5Foreign Key (logical) Reference to another table’s PK. Why: here it’s a logical FK — departmentId stored as a column (see JPA design note). User.java departmentId references departments.departmentId at the application level.
6UNIQUE constraint No duplicate values in a column. Why: two users can’t share an email. @Column(unique=true) email + service guard existsByEmail() → DuplicateEmailException (409).
7Indexes Speed up filtered lookups. Why: user listing filters by role/status often. User.java @Index on role, status, and composite (role, status).
8Transactions (COMMIT/ROLLBACK) All-or-nothing unit of work (ACID). Why: consistency across multi-step writes. Driven by @Transactional service methods (Spring + Hibernate manage commit/rollback).
9createDatabaseIfNotExist JDBC URL convenience flag. Why: first run creates the schema automatically in dev. application.properties: jdbc:mysql://localhost:3306/civicdesk?createDatabaseIfNotExist=true.
10H2 for tests In-memory SQL DB. Why: fast, isolated repository tests without a real MySQL. pom.xml h2 (test) + test/resources/application-test.properties; used by UserRepositoryTest.java.

Bean Validation (real in this project)

Jakarta Validation guards every write endpoint at the boundary — failures become 400s automatically.

#ConceptTheory + Why we use itExact file & how it is used
1@Valid + @RequestBody Triggers validation of the incoming DTO. Why: reject bad input before it hits business logic. UserController.java: createUser(@Valid @RequestBody CreateUserRequest req...).
2Field constraints @NotBlank @Email @Size @Pattern. Why: declarative rules beside the field — readable and reusable. dto/request/CreateUserRequest.java: @Email email, @Pattern(^[6-9][0-9]{9}$) phone, @Pattern(DS|FO|ENG|CO) role.
3Custom constraint Your own annotation + validator. Why: validate a value is a member of an enum, reusably. module/auditlog/validation/EnumValid.java + EnumValidator.java (tested in EnumValidatorTest.java).
4Validation → 400 Invalid args throw MethodArgumentNotValidException. Why: consistent error body listing field messages. GlobalExceptionHandler.java handler joins field errors → ApiResponse.error(...) with 400.

Spring Security + JWT (real in this project)

Stateless JWT auth with role-based access. Expect heavy viva questions here.

#ConceptTheory + Why we use itExact file & how it is used
1SecurityFilterChain Declares what is public vs authenticated. Why: lock down everything except login/register/Swagger. config/SecurityConfig.java: permitAll() for /iam/auth/** + Swagger; anyRequest().authenticated(); STATELESS; CSRF off.
2JWT issue / verify Signed token carrying userId+role. Why: stateless auth — no server session store. common/util/JwtUtil.java — HS256 generateToken(), validateToken(), extractRole(); secret/expiry from JwtConfig.java.
3Auth filter Runs once per request, sets the principal. Why: populate SecurityContext from the Bearer token; sliding refresh near expiry. module/iam/security/JwtAuthFilter.java extends OncePerRequestFilter; registered addFilterBefore(...) in SecurityConfig.
4Method security @PreAuthorize Role check per endpoint. Why: only ADM/DS create users; only ADM changes status. UserController.java: @PreAuthorize("hasAnyRole('ADM','DS')"), @PreAuthorize("hasRole('ADM')"); enabled by @EnableMethodSecurity.
5Password hashing BCrypt one-way hash. Why: never store plaintext passwords. @Bean PasswordEncoder = BCryptPasswordEncoder(10) (SecurityConfig); encode()/matches() in AuthServiceImpl.java.
6CORS Allow specific frontend origins. Why: Angular/React dev servers call the API cross-origin. config/CorsConfig.java reads app.cors.allowed-origins from properties.

Lombok

Honest status: Lombok is in the syllabus but is NOT used in this project — there is no Lombok dependency in pom.xml. All getters/setters/constructors are written by hand (e.g. entity/User.java, dto/response/UserResponse.java). Know the annotations conceptually and be ready to say where they would apply.
#ConceptTheory + Why it existsManual equivalent in this project
1@Data / @Getter / @Setter Generates accessors, toString, equals, hashCode. Why use it: remove boilerplate. Hand-written getters/setters fill ~120 lines of entity/User.java — exactly what @Data would replace.
2@NoArgsConstructor / @AllArgsConstructor Generates constructors. Why use it: JPA needs a no-arg ctor; DTOs need full ctor. Written manually: public User(){} and UserResponse(...) all-args ctor.
3@Builder Fluent object construction. Why use it: readable creation of objects with many fields. Not used; instead the static factory UserResponse.from(user) builds the DTO.
4@Slf4j Injects a logger field. Why use it: skip the boilerplate logger line. Done manually instead: private static final Logger log = LoggerFactory.getLogger(...) (see Logging section).

SLF4J & Logging

Real: SLF4J is used via LoggerFactory.getLogger(...) (the manual form, since Lombok isn’t present).
#ConceptTheory + Why we use itExact file & how it is used
1Logger instantiation SLF4J facade; Logback backend (Boot default). Why: abstraction over the logging impl. config/DataSeeder.java & auditlog/client/AuditClient.java: private static final Logger log = LoggerFactory.getLogger(X.class);
2Log levels TRACE<DEBUG<INFO<WARN<ERROR. Why: INFO for normal flow, WARN for recoverable problems. DataSeeder.java log.info("Seeded department '{}' as '{}'", ...); AuditClient.java log.warn("Failed to record audit entry ...").
3Parameterised logging {} placeholders, lazy formatting. Why: avoids string concatenation cost when level is off. Used in both files above — values passed after the message string. Why use {} instead of "a"+b in a log call?
4SQL logging Hibernate prints generated SQL. Why: debug what queries actually run. application.properties: spring.jpa.show-sql=true.

JUnit 5 & Mockito (TDD)

Three layers of tests under src/test/java: unit (Mockito), slice (@WebMvcTest/@DataJpaTest), and full integration.

#ConceptTheory + Why we use itExact file & how it is used
1@ExtendWith(MockitoExtension) Wires Mockito into JUnit 5. Why: auto-init mocks per test. service/UserServiceImplTest.java top-level annotation.
2@Mock / @InjectMocks Fake collaborators injected into the unit under test. Why: test service logic in isolation, no DB. UserServiceImplTest.java: @Mock UserRepository, @Mock DepartmentRepository, @InjectMocks UserServiceImpl.
3when/thenReturn Stub a mock’s behaviour. Why: drive each branch deterministically. when(userRepository.existsByEmail(anyString())).thenReturn(false) in UserServiceImplTest.java.
4Assertions assertEquals, assertThrows. Why: verify both happy path and error path. assertEquals("DS", res.getRole()); assertThrows(ForbiddenException.class, () -> ...).
5@WebMvcTest Loads only the web layer + MockMvc. Why: fast controller tests without the full context. controller/UserControllerTest.java, AuthControllerTest.java.
6@DataJpaTest Loads only JPA + in-memory H2. Why: test queries/derived methods quickly. repository/UserRepositoryTest.java, auditlog/repository/AuditLogRepositoryTest.java.
7@SpringBootTest (integration) Full application context. Why: end-to-end checks (RBAC, JWT, id generation). iam/integration/RbacIntegrationTest.java, AuthIntegrationTest.java, IdGenerationIntegrationTest.java.
8Validation tests Assert constraint messages fire. Why: lock the API contract for bad input. controller/UserRequestValidationTest.java, AuthRequestValidationTest.java.

Design Patterns & SOLID

#ConceptTheory (one line)Where applied in this project
1S — Single ResponsibilityOne reason to change per class. Controller = HTTP, Service = rules, Repository = data, JwtUtil = tokens only.
2O — Open/ClosedExtend without modifying. New filters added via UserSpecifications factory methods, not by editing existing queries.
3L — Liskov SubstitutionSubtype replaces base safely. UserServiceImpl substitutes UserService wherever injected.
4I — Interface SegregationSmall, focused interfaces. Separate UserService, AuthService, DepartmentService — no fat interface.
5D — Dependency InversionDepend on abstractions. Controllers depend on service interfaces; concrete impl injected by Spring.
6Repository PatternAbstracts data access. repository/UserRepository.java via Spring Data JPA.
7DTO PatternSeparate transfer objects. dto/request/* and dto/response/*.
8Factory MethodStatic method builds objects. UserResponse.from(user), ApiResponse.data/of/error(), PageResponse.from(...).
9SingletonOne shared instance. Every Spring bean (default scope).
10Strategy / SpecificationInterchangeable algorithms/predicates. UserSpecifications.java — composable query predicates.
11Filter / Chain of ResponsibilityRequest passes through filters. JwtAuthFilter.java in the servlet filter chain.
12MVCSeparate control / model / view. Controller → Service → Repository; JSON via ApiResponse is the “view”.

Data Structures & Algorithms

#ConceptTheory + Why we use itExact file & how it is used
1List (immutable) List.of(...) fixed collection. Why: safe constant allow-lists. UserServiceImpl.java SUPERVISOR_CREATABLE_ROLES; DataSeeder.java department list.
2Optional Null-safe container. Why: force callers to handle “not found”. findById(id).orElseThrow(() -> new ResourceNotFoundException(...)) across services.
3Stream pipeline map/filter/mapToInt/max. Why: compute next department id declaratively. DataSeeder.nextDepartmentId() — scans DPT\d+ ids for the max, then increments.
4Pagination (algorithmic) Offset/limit windowing. Why: bounded memory + response size. PageRequest.of(page,size) in UserServiceImpl.getUsers().
5Index-backed lookup DB index ≈ O(log n) vs O(n) scan. Why: push filtering to indexed columns, not in-memory loops. User.java @Index on role/status; filtering via Specifications at the DB. Why filter in SQL instead of fetching all and filtering in Java?

Likely Viva Questions (with the project answer)

#QuestionHow to answer using THIS project
1Walk me through a request.JwtAuthFilter validates the token → sets SecurityContext → @PreAuthorize checks role → UserController validates the DTO → UserServiceImpl (@Transactional) applies rules → UserRepository hits MySQL → ApiResponse JSON returned; errors flow through GlobalExceptionHandler.
2Why constructor injection, not @Autowired fields?Immutable final deps, guaranteed non-null, and easy to construct in unit tests (see UserServiceImplTest with @InjectMocks).
3Why no @ManyToOne between User and Department?Deliberate ID-reference design for loose coupling / future service split, flatter JSON, and no lazy-loading/N+1 surprises. Integrity is enforced in the service layer.
4How is the API stateless?SessionCreationPolicy.STATELESS + JWT in the Authorization header; no server session. Near-expiry tokens are refreshed by JwtAuthFilter.
5How do you keep create-user atomic?@Transactional on createUser() — saving the user and updating the department’s supervisor commit together or roll back together.
6How are validation errors returned?@Valid throws MethodArgumentNotValidException; GlobalExceptionHandler joins field messages into a 400 ApiResponse.
7How do you test without a real DB?Mockito mocks the repository for service unit tests; @DataJpaTest + H2 for repository tests; @SpringBootTest for integration.
8Where are passwords stored?Only the BCrypt hash (passwordHash) is stored; UserResponse never exposes it.

Student note: For every concept, answer in the pattern What → Where (file) → Why. The strongest answers name the exact file (e.g. UserServiceImpl.java) and the reason behind the design choice — that’s what separates “knows the syllabus” from “built the project”.

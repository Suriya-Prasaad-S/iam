1. Executive Summary

1.1 What This Project Is

CivicDesk is a civic-services platform (the kind a municipal corporation uses to handle citizen grievances, service requests, permits, and inspections). This document describes Module 2.1 — Identity & Access Management (IAM), the foundation module on which every other module depends.

The IAM module answers three questions for the entire platform:

  • Who are you? — authentication of citizens and staff (login, registration, password setup).
  • What are you allowed to do? — authorization through role-based access control (RBAC).
  • What did you do? — an append-only audit trail of security-relevant actions.

1.2 Core Capabilities

  • Citizen self-registration — a member of the public creates their own account.
  • Two separate login portals — one for citizens, one for staff, each rejecting the other’s users.
  • JWT issuance — on successful login the server returns a stateless, signed token (HS256, 30-minute validity) that the client presents on every later request.
  • Staff onboarding — administrators create staff accounts without passwords; the staff member sets their own password on first use.
  • Role-based administration — an Admin can create Department Supervisors; a Supervisor can create field staff within their own department.
  • User lifecycle management — activate, suspend, or deactivate accounts.
  • Audit logging — every register / login / logout / user-create / status-change / password-set event is recorded with the actor, action, module, IP address, and timestamp.
  • Department directory — six seeded municipal departments to which staff are assigned.

1.3 How to Read This Document

The document moves from the “big picture” to the smallest detail:

  • Sections 1–3 — what the system does, the technology stack, and the layered architecture (the “why” behind the folder structure).
  • Sections 4–9 — a layer-by-layer walkthrough of every source file: configuration, the shared “common” layer, the data layer (entities/repositories), DTOs, services, and controllers.
  • Section 10 — an in-depth treatment of security and JWT authentication (the heart of an IAM module).
  • Sections 11–14 — end-to-end flows, a complete annotation reference, cross-cutting concerns (errors, logging, configuration), and the testing strategy.

Each file is documented under four lenses: WHAT it does, WHY it exists, HOW it works internally, and HOW it connects to the rest of the system.

3.3 Separation of Concerns — Why This Structure?

ControllerHTTP concerns only (routing, status codes, serialization).Easy to read; swap transport without touching logic.
ServiceBusiness rules and orchestration.Reusable, independently testable, transactional.

This structure means a change in one concern rarely forces a change in another: you can rename a database column without changing the API, add a validation rule without touching the database, or rewrite the security filter without touching business logic.

4. Configuration & Application Bootstrap

This layer wires the application together: it starts the Spring context, defines the security rules, binds configuration properties, configures CORS and Swagger, and seeds baseline data.

4.1 CivicDeskApplication — the Entry Point

File: com/civicdesk/CivicDeskApplication.java

  • WHAT: The Spring Boot entry point. Its main(String[] args) calls SpringApplication.run(CivicDeskApplication.class, args), which starts the Spring context, scans for components, launches the embedded Tomcat server, and runs any startup runners.
  • WHY: Every Spring Boot application needs exactly one bootstrap class to trigger auto-configuration and component scanning.
  • HOW it connects: Because the class sits in the root package com.civicdesk, its component scan covers every sub-package — config, common, and module.iam.*. After startup, the DataSeeder runner executes.

Annotation

  • @SpringBootApplication — a meta-annotation combining three: @Configuration (the class may declare beans), @EnableAutoConfiguration (Spring configures itself based on the classpath — JPA, web, security), and @ComponentScan (discovers all @Component/@Service/@Repository/@RestController/@Configuration beans under com.civicdesk). It is what lets the entire module wire itself up with no XML.

4.2 SecurityConfig — the Heart of Authentication/Authorization

File: com/civicdesk/config/SecurityConfig.java

  • WHAT: The single source of truth for how requests are secured. It defines the HTTP security filter chain (stateless JWT auth), the public-vs-protected URL rules, the placement of the custom JWT filter, the 401 entry point, and the password-encoder bean.
  • WHY: It converts the application into a stateless, token-secured REST API: most endpoints require a valid JWT, a handful of auth/Swagger endpoints are public, and fine-grained role checks happen via @PreAuthorize.

The filterChain(HttpSecurity http) bean — step by step

  1. .cors(Customizer.withDefaults()) — enables CORS, looking up the CorsConfigurationSource bean defined in CorsConfig.
  2. .csrf(csrf -> csrf.disable()) — disables CSRF protection. This is correct for a stateless token API: there are no session cookies to protect, so CSRF tokens are unnecessary.
  3. .sessionManagement(s -> s.sessionCreationPolicy(STATELESS)) — the server keeps no HTTP session; every request must carry its own JWT.
  4. .authorizeHttpRequests(...) — rules evaluated top-down:
    • permitAll() for /iam/auth/register, /iam/auth/citizen/login, /iam/auth/staff/login, /iam/auth/setPassword (these are how a user obtains a token).
    • permitAll() for Swagger paths (/swagger-ui.html, /swagger-ui/**, /v3/api-docs, /v3/api-docs/**).
    • .anyRequest().authenticated() — every other endpoint requires a valid authenticated principal.
  5. .exceptionHandling(...) — a custom AuthenticationEntryPoint returns a clean HTTP 401 when a protected endpoint is hit without a valid token (instead of the default servlet error page).
  6. .addFilterBefore(jwtAuthFilter, UsernamePasswordAuthenticationFilter.class) — inserts the custom JwtAuthFilter before Spring’s username/password filter, so the request is already authenticated from its token by the time authorization runs.
  7. return http.build(); — builds the SecurityFilterChain bean.

The passwordEncoder() bean

Returns new BCryptPasswordEncoder(10) — a BCrypt encoder with cost factor 10. It is exposed as a PasswordEncoder bean and reused by DataSeeder (to hash the seeded admin) and the auth/user services (to hash on write and verify on login).

Annotations

  • @Configuration — marks the class as a source of bean definitions.
  • @EnableMethodSecurity — turns on annotation-based method security (@PreAuthorize, etc.). Without it, those annotations on controllers would be silently ignored. (Modern replacement for @EnableGlobalMethodSecurity.)
  • @Bean — registers the two methods’ return values as managed beans.

4.3 JwtConfig — Typed JWT Settings

File: com/civicdesk/config/JwtConfig.java

  • WHAT: A typed configuration holder binding the app.jwt.* properties: secret (the HS256 signing secret) and expiry (token lifetime in milliseconds).
  • WHY: Centralizes JWT settings into one strongly-typed object rather than scattering @Value lookups; components that mint/verify tokens inject this single bean.
  • HOW it connects: Consumed by JwtUtil for signing/verifying tokens. The keys live in application.properties (app.jwt.secret, app.jwt.expiry).

Annotations

  • @Configuration — registers it as a bean.
  • @ConfigurationProperties(prefix = "app.jwt") — binds all properties under app.jwt to fields by name (type-safe, grouped configuration).

4.4 CorsConfig — Cross-Origin Policy

File: com/civicdesk/config/CorsConfig.java

  • WHAT: Defines the CORS policy so a browser-based frontend on a different origin/port can call this backend.
  • WHY: Browsers block cross-origin requests by default; this config explicitly allows the frontend origins, methods, headers, and credentials.
  • HOW: A @Value("${app.cors.allowed-origins:http://localhost:3000,http://localhost:5173}") field supplies a comma-separated origin list (defaulting to common React/Vite dev ports). The corsConfigurationSource() bean builds a CorsConfiguration that allows those origins; methods GET, POST, PUT, PATCH, DELETE, OPTIONS (OPTIONS for preflight); all headers; exposes the Authorization response header (so the frontend can read a refreshed token); and enables credentials. It is registered for all paths (/**).
  • HOW it connects: This bean is exactly what SecurityConfig’s .cors(Customizer.withDefaults()) picks up.

4.5 OpenApiConfig — Swagger / API Documentation

File: com/civicdesk/config/OpenApiConfig.java

  • WHAT: Configures the OpenAPI metadata (title, description, version) and registers a global bearer (JWT) security scheme so Swagger’s “Authorize” button lets you paste a token and call secured endpoints.
  • HOW: The civicDeskOpenAPI() bean returns an OpenAPI object with .info(...), a global SecurityRequirement named bearerAuth, and a matching SecurityScheme of type HTTP / scheme bearer / format JWT.
  • HOW it connects: The Swagger paths it enables are precisely the ones SecurityConfig marks permitAll(); the bearer scheme corresponds to the Authorization: Bearer <token> header validated by JwtAuthFilter.

4.6 DataSeeder — Startup Bootstrap Data

File: com/civicdesk/config/DataSeeder.java (implements CommandLineRunner)

  • WHAT: A startup runner that seeds the six municipal departments and the very first admin account directly into the database — bypassing the API, since no admin exists yet to call protected endpoints.
  • WHY: Bootstraps an empty database with the baseline data the system needs: departments to assign staff to, and a first administrator who can then create everyone else.
  • HOW:
    • run(String... args) — the CommandLineRunner callback; calls seedDepartments() then seedAdmin().
    • seedDepartments() — for each of the six fixed names (Infrastructure, Public Health, Licensing & Compliance, Citizen Services, Administration, Compliance & Audit), if it does not already exist, creates it with a generated id. Idempotent.
    • nextDepartmentId() — produces sequential ids DPT01, DPT02, … by taking the max existing numeric suffix + 1 (zero-padded).
    • seedAdmin() — if no user with the admin email exists, builds a User with role ADM, status active, BCrypt-hashed password (from app.admin.password), passwordSet = true, no department; saves it; then records a SEED_ADMIN audit entry. Idempotent.
  • HOW it connects: Runs automatically because SpringApplication.run executes all CommandLineRunner beans. Uses the PasswordEncoder bean from SecurityConfig, the IAM entities/enums/repositories, and AuditService. The seeded admin is the first principal able to authenticate and call the @PreAuthorize-protected endpoints.
  • Annotations: @Component (discovered by component scan; its run executes at startup); @Value on the admin fields (injects app.admin.*, which are environment-overridable).

5. The Common Layer (Shared Building Blocks)

The common package holds module-agnostic infrastructure: a unified response envelope, a centralized exception handler with a family of domain exceptions, security/utility helpers, and a custom ID generator. These are shared across every module of the platform.

5.1 Response Envelopes (common/response)

5.1.1 ApiResponse

File: common/response/ApiResponse.java

  • WHAT: A unified JSON response envelope with two fields, message and data, used for both success and error responses.
  • WHY: Gives every endpoint a single, predictable shape. Null fields are suppressed, so the same class produces clean payloads for reads (data only), writes (message + optional data), and errors (message only).
  • HOW — factory methods:
    • ApiResponse.data(Object data) — for GET success (data only).
    • ApiResponse.of(String message, Object data) — for POST/PUT success (message + optional data).
    • ApiResponse.error(String message) — for error responses (message only).
  • Annotation: @JsonInclude(JsonInclude.Include.NON_NULL) (Jackson) — omits any null field during serialization, which is what makes message/data disappear when unset.
  • HOW it connects: Returned by all IAM controllers and produced by GlobalExceptionHandler for errors.

5.1.2 ErrorResponse

File: common/response/ErrorResponse.java

  • WHAT: A richer, path-aware error envelope carrying status, message, path, and timestamp; a static of(...) factory stamps the timestamp automatically.
  • WHY / connection: Provided for cross-module consistency. The IAM GlobalExceptionHandler currently uses ApiResponse instead, so this type is available for other modules that prefer the richer shape.
  • Annotation: @JsonPropertyOrder({"status","message","path","timestamp"}) — fixes the JSON field order for deterministic output.

5.1.3 PageResponse<T>

File: common/response/PageResponse.java

  • WHAT: A lightweight, framework-neutral pagination envelope: content, totalElements, totalPages, currentPage.
  • WHY: Decouples the public API from Spring Data’s internal Page type, keeping the wire format stable and minimal.
  • HOW: The key factory static <S,T> PageResponse<T> from(Page<S> page, Function<S,T> mapper) builds the envelope from a Spring Data Page, converting each entity to a response DTO via the mapper and copying the totals and current (zero-based) page number.
  • HOW it connects: Used by the user-listing and audit-log endpoints; the mapper typically converts entity → DTO (e.g. UserResponse::from).

5.2 Exception Handling (common/exception)

5.2.1 GlobalExceptionHandler

File: common/exception/GlobalExceptionHandler.java

  • WHAT: A centralized, application-wide handler that catches exceptions thrown anywhere in the controllers/services and converts each into an ApiResponse.error(...) body with a specific HTTP status.
  • WHY: Keeps controllers free of try/catch and guarantees a consistent error envelope and status code for every error.
  • Annotations: @RestControllerAdvice (global advice for all controllers; combines @ControllerAdvice + @ResponseBody), and one @ExceptionHandler(Type.class) per method.

The thoughtful split between AccountSuspendedException (423) and AccountInactiveException (403) lets clients react differently to a locked vs a merely inactive account.

5.3 Utilities (common/util)

5.3.1 JwtUtil — Token Authority

File: common/util/JwtUtil.java — covered in depth in Section 10. In summary, it is a @Component that issues and verifies HS256 JWTs using the JJWT library, injecting JwtConfig for the secret and expiry. Key methods: generateToken(userId, role), validateToken(token), extractClaims/extractUserId/extractRole/extractExpiration.

5.3.2 NationalIdUtil — PII Protection

File: common/util/NationalIdUtil.java (final, non-instantiable static helper)

  • WHAT/WHY: Protects a citizen’s national ID. The raw ID is never stored — only a one-way hash — and only the last four digits are ever displayed.
  • HOW: hash(nationalId) computes a SHA-256 digest and hex-encodes it (returns null for blank input). maskLast4(nationalId) returns the value with all but the last four characters replaced by * (e.g. ********1234).

5.3.3 SecurityContextUtil — Reading the Current User

File: common/util/SecurityContextUtil.java (final, non-instantiable)

  • WHAT/WHY: A clean, one-line way for service/controller code to read the currently authenticated caller without touching Spring Security internals. It reads exactly what JwtAuthFilter placed into the context.
  • HOW: getCurrentUserId() returns the principal (the JWT userId); getCurrentRole() returns the first authority with the ROLE_ prefix stripped.
  • HOW it connects: Used by controllers to pass the caller’s id/role into services, and by audit logging to record who acted.

5.3.4 ClientIpUtil — Resolving the Client IP

File: common/util/ClientIpUtil.java (final, non-instantiable)

  • WHAT/WHY: Resolves the true client IP for audit records, accounting for proxies/load balancers and IPv6 loopback.
  • HOW: resolve(request) prefers the first entry of the X-Forwarded-For header, falling back to getRemoteAddr(); it normalizes IPv6 loopback (::1) to 127.0.0.1 and returns "UNKNOWN" when nothing is available.
  • HOW it connects: Controllers call it and pass the result into AuditService.log(...).

5.4 Custom ID Generation (common/id)

NumericStringSequenceGenerator

File: common/id/NumericStringSequenceGenerator.java (extends Hibernate’s SequenceStyleGenerator)

  • WHAT: A custom Hibernate ID generator that produces sequential numeric ids (10000001, 10000002, …) from a database sequence but stores them in a String primary-key column.
  • WHY: The whole codebase treats user/audit ids as opaque Strings (the JWT userId claim, the security principal, repository keys, DTOs). Keeping the id a String avoids a type change rippling through every layer, while giving human-friendly running numbers instead of UUIDs. It works on both H2 (native sequences, used in tests) and MySQL (Hibernate emulates with a backing table).
  • HOW: configure(...) tells the parent generator the type is Long/BIGINT (so it can drive the sequence) even though the entity field is a String; generate(...) fetches the next numeric value and converts it with String.valueOf(...) before it is assigned to the @Id field.
  • HOW it connects: Referenced by the User and AuditLog entity @Id mappings; the strings it mints become the JWT userId and the Spring Security principal.

6. The Data Layer: Entities, Enums, Repositories & Schema

6.1 Cross-Cutting Persistence Design

  • Naming strategy: the app uses PhysicalNamingStrategyStandardImpl, so Hibernate keeps identifiers as-is (camelCase columns like userId, passwordHash, createdAt) rather than converting to snake_case.
  • ID generation: User and AuditLog use the custom NumericStringSequenceGenerator (sequential numeric strings from 10000001); Department uses hand-assigned DPT01–style keys.
  • Enums stored as strings: entity columns role, status, action, module are plain String (no @Enumerated). The enums exist for application-layer validation/normalization; the database sees raw strings, which keeps the values stable and module-shareable.

  • Class annotations: @Entity (JPA-managed persistent class), @Table(name="users", indexes={...}) declaring three indexes — idx_users_role, idx_users_status, and the composite idx_users_role_status (supporting the common combined filter).

6.4 Entity: Department

File: module/iam/entity/Department.java

  • WHAT/WHY: Maps the departments table — the organizational units that group staff and track the heading supervisor; provides a stable, human-readable code.

7. Data Transfer Objects (DTOs)

7.1 Why DTOs Instead of Exposing Entities

DTOs form the contract between the API and its clients. They exist separately from JPA entities for three reasons:

  • Security: entities carry sensitive/internal fields (e.g. passwordHash, internal flags). Returning them risks leaking secrets, and binding requests straight onto them risks “mass-assignment” attacks (a client setting a field it shouldn’t, like its own role). UserResponse deliberately omits the password hash.
  • Decoupling: the wire format can evolve independently of the database. Renaming a column does not break the API, and vice versa.
  • Validation & performance: request DTOs are the boundary where input is validated (rejecting malformed data with 400 before it reaches business logic), and they carry only the fields actually needed (no over-fetching of entity graphs).
Implementation note: these DTOs are plain POJOs with hand-written getters/setters and explicit constructors — Lombok is not used in this codebase. (Lombok’s @Data/@Builder etc. are explained generally in the annotation reference for completeness.) Response DTOs each provide a static from(entity) factory for entity→DTO mapping.

  • @NotBlank — the string must be present and not just whitespace.
  • @NotNull — the value must not be null (used for non-string fields).

Using @Pattern whitelists for role and status gives clean, human-readable 400 messages instead of cryptic deserialization errors. Validation only fires because the controllers annotate the body with @Valid.

8. The Service Layer (Business Logic)

8.1 Why a Service Layer Exists

The service layer sits between the web layer (controllers) and the persistence layer (repositories). Its purpose is separation of concerns:

  • Business-logic isolation — rules like “an Admin may only create a Department Supervisor”, password validation, status normalization, and role-based filtering live here, not in controllers, making them reusable and independently testable.
  • Transaction boundaries — multi-step writes are wrapped in @Transactional so they commit or roll back atomically.
  • Abstraction via interfaces — each service is an interface plus a @Service implementation, so controllers depend on the abstraction and tests can mock it.

Each service follows the pattern: an interface (AuthService, UserService, AuditService, DepartmentService) and an implementation (...Impl). All use constructor injection with final fields (no field @Autowired).

8.2 AuthServiceImpl

Files: service/AuthService.java (interface), service/AuthServiceImpl.java

WHAT/WHY: Implements registration, citizen/staff login, and first-time password setup. It centralizes all credential verification, account-status gating, password hashing, and token issuance so controllers stay thin. Dependencies: UserRepository, PasswordEncoder, JwtUtil, AuditService.

Methods, step by step

  • register(req, ip) — @Transactional. Rejects a duplicate email (DuplicateEmailException); builds a citizen User; BCrypt-hashes the password; sets role CIT, active status, passwordSet=true; saves; writes a REGISTER audit entry.
  • citizenLogin(req, ip) — calls the shared authenticate(...); if the user is not a citizen, throws ForbiddenException("Please use the staff portal"); otherwise issues a token.
  • staffLogin(req, ip) — calls authenticate(...); if the user is a citizen, throws ForbiddenException("Please use the citizen portal"); otherwise issues a token.
  • authenticate(email, password) (private) — the shared credential + status gate: looks up by email (else BadCredentialsException); if !passwordSet throws PasswordNotSetException; verifies the password with passwordEncoder.matches(...) (else BadCredentialsException with the same generic message to prevent user enumeration); normalizes status and throws AccountSuspendedException (suspended) or AccountInactiveException (not active).
  • setPassword(req) — @Transactional. Looks up the user (else ResourceNotFoundException); if a password is already set throws ForbiddenException (one-time only); enforces minimum length; BCrypt-hashes, sets passwordSet=true, saves; writes a SET_PASSWORD audit entry.
  • issueToken(user, ip) (private) — generates a JWT via jwtUtil.generateToken(userId, role); writes a LOGIN audit entry; returns an AuthResponse with the token, id, role, and expiry in seconds.

Connections: UserRepository, PasswordEncoder, JwtUtil, AuditService; the Role/UserStatus/AuditAction enums; and the domain exceptions. Annotations: @Service; @Transactional on the two write methods; @Value("${app.jwt.expiry}") for the expiry.

9. The Controller Layer (REST API)

9.1 Role of Controllers & the “Thin Controller” Principle

Controllers are the HTTP front door. In a REST API they map URLs and HTTP verbs to Java methods, read request bodies/parameters, and serialize responses to JSON. In this codebase every controller method does only orchestration:

  1. Read the caller’s identity/role from the security context (SecurityContextUtil).
  2. Resolve the client IP (ClientIpUtil) when auditing.
  3. Delegate the real work to a service method.
  4. Optionally fire an audit log.
  5. Wrap the result in ApiResponse / ResponseEntity.

There is no business logic, persistence, or mapping in controllers — methods are typically 3–6 lines. This keeps them focused on HTTP concerns and makes the business layer independently testable. DepartmentController is the clearest example: a single one-line delegation.

9.2 Common Controller Annotations

  • @RestController — a controller whose return values are written directly to the HTTP body as JSON (combines @Controller + @ResponseBody).
  • @RequestMapping("/path") — the shared base path for all methods in the class.
  • @GetMapping / @PostMapping / @PutMapping — map a method to HTTP GET / POST / PUT at a sub-path.
  • @PathVariable — binds a {placeholder} URL segment to a parameter.
  • @RequestParam — binds a query-string parameter (supports required=false and defaultValue).
  • @RequestBody — deserializes the JSON body into a DTO.
  • @Valid — triggers Bean Validation on the DTO before the method runs.
  • @PreAuthorize("...") — a method-level security gate using SpEL (hasRole('ADM'), hasAnyRole('ADM','DS')). Spring matches the ROLE_-prefixed authority the JWT filter installs.
  • Constructor injection — dependencies are final fields set through the constructor (no @Autowired needed for a single constructor).

10. Security & JWT Authentication (In Depth)

This is the core of an IAM module. This section explains JWT conceptually, then walks through every line of the code that issues, transmits, validates, and authorizes with tokens.

10.1 What Is a JWT?

A JSON Web Token (JWT) is a compact, URL-safe, self-contained credential. “Self-contained” means the token itself carries the identity facts (claims) the server needs — the server does not look anything up in a session store to know who the caller is. In CivicDesk, a successful login produces a JWT, and the client sends it back on every request as the header Authorization: Bearer <token>.

10.2 The Three Parts of a JWT

A JWT is three Base64URL segments joined by dots: header.payload.signature.

  eyJhbGciOiJIUzI1NiJ9 . eyJ1c2VySWQiOiIxMDAwMDAwMSIsInJvbGUiOiJBRE0iLCJpYXQiOi4uLn0 . 3x9Kk2...signature
  └──── HEADER ────┘   └──────────────── PAYLOAD (claims) ────────────────┘   └─── SIGNATURE ───┘
  • Header — JSON metadata about the token, notably the signing algorithm. JJWT writes {"alg":"HS256"}.
  • Payload (claims) — the data. This codebase puts:
    • userId — the user’s id → becomes the Spring Security principal.
    • role — the user’s role → becomes the ROLE_<role> authority.
    • iat (issued-at) and exp (expiration, 30 minutes out).
    Claims are only Base64URL-encoded, not encrypted — anyone can read them, so no secrets are placed there.
  • Signature — proves the token was issued by this server and has not been altered.

10.3 How HS256 Signing Works

HS256 = HMAC with SHA-256, a symmetric scheme. The server computes:

signature = HMAC_SHA256( base64url(header) + "." + base64url(payload), SECRET )

using the single shared secret from app.jwt.secret. On every request, validation recomputes the HMAC over the received header.payload with the same secret and checks it matches the token’s signature. If even one byte of the payload changed (e.g. an attacker editing role to gain admin), the recomputed signature would not match and verification fails. Because the same secret signs and verifies, it must stay server-side only — HS256 also requires the secret to be at least 256 bits / 32 bytes, which is why the configured secret is “minimum_32_characters”.

10.4 Token Generation & Verification — JwtUtil

File: common/util/JwtUtil.java (@Component, injects JwtConfig)

  • Signing key — getKey() derives an HMAC-SHA key from the UTF-8 bytes of the secret via Keys.hmacShaKeyFor(...).
  • generateToken(userId, role) — builds the token: .claim("userId", userId), .claim("role", role), .issuedAt(now), .expiration(now + expiry), .signWith(getKey()) (HS256 inferred), .compact().
  • validateToken(token) — parses and verifies signature/structure/expiry; returns true on success, false on any exception (bad signature, malformed, expired).
  • extractClaims / extractUserId / extractRole / extractExpiration — verify and read the payload; unlike validateToken, these throw on an invalid/expired token. extractExpiration feeds the sliding-expiry refresh.

10.5 The Per-Request Gatekeeper — JwtAuthFilter

File: module/iam/security/JwtAuthFilter.java (extends OncePerRequestFilter)

Why OncePerRequestFilter? It guarantees the filter runs exactly once per HTTP request even across internal forwards/async dispatches — you do not want to re-parse the token or re-populate the context multiple times.

doFilterInternal(req, res, chain) — step by step

  1. Read the Authorization header.
  2. If it is missing or does not start with "Bearer ", pass through unauthenticated (chain.doFilter) — this is what lets the permitAll() endpoints work without a token; protected routes are rejected later by the 401 entry point.
  3. Strip the "Bearer " prefix to get the raw token.
  4. If !jwtUtil.validateToken(token), set status 401 and stop — an invalid/expired/tampered token is rejected immediately.
  5. Extract role and userId from the verified token.
  6. Build a UsernamePasswordAuthenticationToken with principal = userId, credentials = null, and a single authority "ROLE_" + role (the ROLE_ prefix is what hasRole(...) expects). The 3-argument constructor marks it authenticated.
  7. Store it in SecurityContextHolder — the caller is now authenticated for the rest of the request.
  8. Sliding-expiry refresh: compute seconds remaining from the exp claim; if ≤ 600 (10 minutes), mint a fresh 30-minute token and return it on the response Authorization header, plus Access-Control-Expose-Headers: Authorization so browsers let JavaScript read it. Active users stay logged in without re-authenticating, with no server-side state.
  9. Continue the chain to the controller.
Token stateBehavior
Missing / not “Bearer ”Pass through unauthenticated; protected routes 401 via the entry point.
Present but invalid / expired / tamperedImmediate 401; chain stops.
ValidAuthenticate, optionally refresh, continue.

Annotation: @Component (so SecurityConfig can inject it). It is wired explicitly with addFilterBefore(...) rather than auto-registered.

10.6 Why JWT Is Stateless (and Why It Matters Here)

Because all identity data and a tamper-proof signature travel inside the token, the server keeps no session record. That is why SecurityConfig uses SessionCreationPolicy.STATELESS and JwtAuthFilter rebuilds the Authentication from scratch on every request. Any server instance holding the shared secret can verify any token — no sticky sessions and no shared session store, which makes horizontal scaling trivial.

10.7 The Authorization Process (RBAC)

  • The filter installs a ROLE_<role> authority from the token’s role claim.
  • @EnableMethodSecurity (in SecurityConfig) activates @PreAuthorize on controller methods.
  • Each protected endpoint declares its rule: hasRole('ADM') (status updates), hasAnyRole('ADM','DS') (user management, departments), hasAnyRole('ADM','CO') (audit logs).
  • If the role does not satisfy the rule, Spring throws AccessDeniedException, which GlobalExceptionHandler turns into a clean 403.
  • Finer business rules that a single role check cannot express (e.g. “a DS may only see their own department”) are enforced in the service layer using the caller’s id/role.

10.8 Advantages & Limitations of This Design

AdvantagesLimitations
No session store → easy horizontal scaling.Hard to revoke before expiry — a leaked token is valid until exp (up to 30 min); there is no server-side blacklist.
Decoupled & reusable — every module trusts the same userId/role claims (JwtUtil is shared across teams).Symmetric secret — the same secret signs and verifies, so a leak lets attackers forge tokens (RS256/ES256 would avoid sharing the signing key).

11. End-to-End Flows

11.1 Generic Request Flow (API call → Controller → Service → DB → Response)

1. Client sends an HTTP request with  Authorization: Bearer <jwt>
2. CORS filter            → applies the allowed-origins policy (CorsConfig)
3. JwtAuthFilter          → validates the token, sets the SecurityContext (principal + ROLE_x)
4. Authorization          → URL rule (authorizeHttpRequests) + @PreAuthorize role check
5. Controller method      → reads body/params (@RequestBody/@RequestParam), @Valid runs
6. Service method         → business rules, calls repositories, hashes/issues tokens, audits
7. Repository (JPA)       → Hibernate generates SQL against MySQL/H2
8. Entity ↔ DTO           → service maps entities to response DTOs (e.g. UserResponse.from)
9. Controller             → wraps the DTO in ApiResponse  →  Jackson serializes to JSON
10. (any exception)       → GlobalExceptionHandler maps it to {message} + the right status
11. Response returned     → standard envelope { success?, statusCode?, message, data, timestamp }

11.4 Data Flow Between Layers

  • Inbound: JSON → request DTO (validated by @Valid) → service → entity → repository → database.
  • Outbound: database → repository → entity → service maps to response DTO (from(...)) → wrapped in ApiResponse/PageResponse → JSON.
  • Entities never cross the controller boundary, so the database shape never leaks to clients.

13. Error Handling, Logging & Configuration

13.1 Error-Handling Strategy

  • Domain exceptions express business failures (e.g. DuplicateEmailException, AccountSuspendedException). Business code throws these without knowing about HTTP.
  • One central handler (@RestControllerAdvice) maps each exception to a specific HTTP status and a uniform ApiResponse.error(message) body (see Section 5.2).
  • Validation failures are handled too: a failed @Valid body produces a 400 with all field errors joined into one message.
  • Security failures are consistent: 401 for missing/invalid tokens (filter + entry point), 403 for failed role checks (AccessDeniedException).
  • Anti-enumeration: login uses the same “Invalid email or password” for both unknown-email and wrong-password, so attackers cannot probe which emails exist.

13.2 Logging Practices

  • Application logging uses SLF4J (e.g. DataSeeder logs seeded departments/admin). SQL logging is controlled per profile (spring.jpa.show-sql, logging.level.org.hibernate.SQL).
  • Audit logging is a distinct, business-level concern persisted to the audit_log table (not a log file): who did what, in which module, from which IP, when. It is queryable via GET /iam/auditLogs for compliance.

14. Testing (JUnit & Mockito)

14.1 Purpose of Testing & the Test Pyramid

Automated tests verify behaviour, catch regressions, document expected behaviour (test names read like a spec), and let the team refactor with confidence. This module layers its tests into the classic pyramid:

  • Unit tests — one class in isolation with mocked dependencies (fast, narrow).
  • Slice tests — load just one layer: @WebMvcTest (controllers) or @DataJpaTest (repositories against H2).
  • Integration tests — the full Spring context + real H2 + real security filter chain (broad, end-to-end).

14.3 Mockito & Why Mocking Is Required

A mock is a scripted stand-in for a real collaborator. Using real databases/encoders/HTTP objects in a unit test would be slow, non-deterministic, and would blur which class actually failed. Mocking lets you isolate one unit, drive every branch (force a repository to return empty to hit a “not found” path), run fast with no I/O, and verify side-effects. Key patterns used:

  • when(dep.method(args)).thenReturn(value) — stub a return value.
  • verify(mock).method(args) / verify(mock, never())... / verifyNoInteractions(mock) — assert (or refute) that calls happened.
  • Argument matchers any(), anyString(), eq(value).
  • ArgumentCaptor — capture the object passed into a mock and inspect it (used to verify the AuditLog built inside the service).

14.5 Unit vs Integration — A Deliberate Division of Labour

The controller slice tests intentionally defer @PreAuthorize 403 enforcement to RbacIntegrationTest, because @WebMvcTest slices do not reliably enforce method security, but the full-stack tests (with the real filter chain) do. This is a good illustration of choosing the right test type for each guarantee.
